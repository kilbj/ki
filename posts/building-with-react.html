用 React + TypeScript 构建可维护的应用

2025-02-22 · 10 min · React

React 与 TypeScript 的组合已经成为现代前端开发的事实标准。这篇文章分享我在实际项目中积累的一些最佳实践。

组件设计原则

好的组件应该是内聚的、可复用的、边界清晰的。以下是我遵循的几个原则：

  • 单一职责 — 一个组件只做一件事
  • 接口优先 — 先定义 Props 类型，再实现组件
  • 避免过早抽象 — 三次重复之前不要提取公共组件
// 好的做法：明确 Props 接口
interface ButtonProps {
  variant: 'primary' | 'secondary' | 'ghost';
  size: 'sm' | 'md' | 'lg';
  children: React.ReactNode;
  onClick?: () => void;
  disabled?: boolean;
}

function Button({ variant, size, children, ...rest }: ButtonProps) {
  return (
    <button className={`btn btn--${variant} btn--${size}`} {...rest}>
      {children}
    </button>
  );
}

状态管理策略

不是所有状态都需要放到全局 store 中。我的分层策略是：

  1. 组件状态 — useState / useReducer，只在组件内部使用
  2. 提升状态 — 通过 Props 下放到子组件
  3. Context — 跨多层组件共享的主题、语言等
  4. 外部 Store — Zustand / Jotai，用于真正的全局状态
"状态应该尽可能靠近它被使用的地方" —— 这是我在状态管理中学到的最重要的一课。

TypeScript 技巧

在 React 项目中使用 TypeScript，有几个模式特别实用：

//  discriminated unions 处理多状态组件
type State<T> =
  | { status: 'idle' }
  | { status: 'loading' }
  | { status: 'success'; data: T }
  | { status: 'error'; error: string };

function useAsync<T>() {
  const [state, setState] = useState<State<T>>({ status: 'idle' });
  // ... 类型守卫让每个分支的数据访问都是安全的
}

目录结构

推荐按功能（Feature）而不是按类型（Type）组织文件：

src/
├── features/
│   ├── auth/
│   ├── blog/
│   └── settings/
├── shared/
│   ├── components/
│   ├── hooks/
│   └── utils/
└── app.tsx

这种结构让每个功能模块自包含，修改一个功能不需要在多个目录之间跳转。
